# Learning Science Reference

yoda 스킬의 모든 출력을 지배하는 학습과학 원칙 레퍼런스.

---

## 4대 필수 원칙 (모든 출력에 항상 적용)

### 1. Before/After/Why 트리플

> 모든 발견 사항에 **문제코드(Before) + 수정코드(After) + 이유(Why)**를 인접 배치한다.

**근거:** Sweller Cognitive Load Theory, Mayer 인접 원칙 (Noetel 2022 메타분석에서 가장 큰 효과 크기), Dunlosky(2013) 정교화 질문.

#### Before 블록

- 실제 문제 코드를 **그대로** 인용한다.
- 축약하거나 의사코드로 대체하지 않는다.
- 문제 지점에 `// ⚠️` 마커를 붙여 시선을 유도한다.

#### Why 블록

- **호기심 트리거**(정보 격차 질문)로 시작한다.
- 원칙을 **명명**한다 (예: "SRP 위반", "경합 조건").
- 실제 영향을 **구체적으로** 설명한다 (크래시율, 성능 수치, 유지보수 비용).

#### After 블록

- 수정 코드 + **핵심 변경에 인라인 주석**을 포함한다.
- 변경과 무관한 코드는 최소화한다.

#### 인접 배치 규칙

- Before, Why, After 세 블록이 **스크롤 없이 함께** 보여야 한다.
- 하나의 발견 사항이 한 화면을 넘기면 분할한다.
- 긴 코드 블록은 핵심 부분만 발췌하고 `// ... (변경 없음)` 으로 생략한다.

### 2. 신호/라벨링

> 모든 발견 사항에 심각도 라벨을 부착한다.

**근거:** Noetel(2022) 신호/큐잉 효과, Conventional Comments.

| 라벨 | 의미 | 사용 기준 |
|------|------|----------|
| 🔴 Must Fix | 반드시 수정 | 크래시, 데이터 손실, 보안 취약점, 심각한 성능 저하 |
| 🟡 Should Improve | 개선 권장 | 유지보수성 저하, 잠재적 버그, 설계 원칙 위반 |
| 🔵 Nit | 사소한 개선 | 네이밍, 포매팅, 컨벤션 불일치 |
| 🟢 Praise | 칭찬 | 잘 작성된 코드, 모범 사례 적용 |
| 💡 Insight | 통찰 | 학습 포인트, 대안 접근법, 팀 차원 논의 제안 |

#### 적용 규칙

- 라벨은 발견 사항 제목의 **맨 앞**에 배치한다: `🔴 Must Fix: 강제 언래핑으로 인한 크래시 위험`
- 출력 상단에 **심각도별 요약 카운트**를 표시한다: `🔴 2 | 🟡 3 | 🔵 1 | 🟢 2 | 💡 1`
- 🟢 Praise는 전체 발견의 최소 20%를 유지하여 긍정적 강화를 제공한다.
- 라벨 없는 발견 사항을 출력하지 않는다.

### 3. 일관성 -- 불필요 제거

> 독자가 이미 아는 것, 발견 사항과 관련 없는 정보를 제거한다.

**근거:** Mayer(2025) 메타분석 -- 불필요한 정보가 학습 효과를 유의미하게 저해함.

#### 제거 대상

- **언어 기초 설명**: 대상 언어의 기본 문법/개념 설명 금지 (독자가 해당 언어 개발자임을 전제).
- **발견과 무관한 코드 컨텍스트**: 리뷰 대상이 아닌 주변 코드를 장황하게 인용하지 않는다.
- **반복 설명**: 동일 원칙이 여러 발견에 적용되면 첫 등장 시 상세 설명 후 이후는 참조(`위 #2 참조`)로 대체.
- **장식적 문구**: "이제 살펴보겠습니다", "다음으로 넘어가서" 등 전환 문구 제거.
- **면책 조항**: "이것은 제 의견입니다", "상황에 따라 다를 수 있습니다" 등 모호한 단서 제거.

#### 보존 대상

- Before/After/Why 트리플의 각 요소는 축약하지 않는다.
- 심각도 라벨과 근거는 항상 유지한다.
- 호기심 트리거 질문은 제거하지 않는다.

---

### 4. 청킹

> 한 섹션/슬라이드/카드 = 한 개념.

**근거:** Miller(1956) 7±2 법칙, Hermans(2021) 프로그래머 인지 부하 연구.

#### 적용 규칙

- 하나의 발견 사항(Before/After/Why)에 **하나의 개념**만 다룬다.
- 하나의 코드 블록에서 여러 문제를 발견하면 **문제별로 분리**한다.
- 전체 출력에서 발견 사항은 **7개 이하**로 유지한다 (7개 초과 시 우선순위별 그룹으로 분할).
- 섹션 간 시각적 구분(수평선, 헤딩)을 명확히 한다.
- 관련 발견 사항은 **그룹 헤딩** 아래 묶되, 그룹 내에서도 각각 독립적으로 이해 가능해야 한다.

## 포맷별 추가 원칙

4대 필수 원칙은 항상 적용하되, 출력 포맷에 따라 아래 원칙을 추가 적용한다.

### md (Markdown)

| 원칙 | 적용 방법 |
|------|----------|
| **이중 코딩** (Mayer) | Mermaid 다이어그램으로 구조/흐름을 시각화. 텍스트 설명과 다이어그램을 병행 |
| **정교화 질문** (Dunlosky 2013) | 각 섹션 끝에 "생각해볼 점" 질문 배치. 독자가 능동적으로 사고하도록 유도 |

### web (HTML)

| 원칙 | 적용 방법 |
|------|----------|
| **점진적 공개** (Nielsen) | 접기(details/summary), 탭으로 심화 내용 숨김. 첫 화면은 핵심만 노출 |
| **인출 연습** (Roediger & Karpicke 2006) | 인터랙티브 퀴즈/체크리스트로 독자가 학습 내용을 직접 떠올리게 함 |
| **개인화** (Mayer) | 대화체 사용, 독자의 코드를 직접 예시로 활용 |

### wiki (Confluence 등)

| 원칙 | 적용 방법 |
|------|----------|
| **스토리텔링** (내러티브) | 문제 발견 → 원인 탐색 → 해결의 서사 구조. "우리 팀이 겪은 실제 사례"로 시작 |
| **사회적 구성주의** (Vygotsky) | 댓글/피드백 유도 질문 배치. "이 부분에 대해 다른 접근법이 있다면 댓글로 공유해주세요" |
| **정교화 질문** (Dunlosky 2013) | "이 패턴을 우리 프로젝트의 다른 모듈에 적용한다면?" 형태로 확장 질문 |

### slides (프레젠테이션 마크다운)

| 원칙 | 적용 방법 |
|------|----------|
| **신호 강화** | 슬라이드 제목에 핵심 메시지 집약. 시각적 강조(볼드, 컬러) 극대화 |
| **일관성 극대화** | 한 슬라이드 = 한 메시지. 불필요한 텍스트/장식 완전 제거 |
| **인지적 도제 모델링** (Collins et al.) | 전문가 사고 과정을 단계별로 노출: "내가 이 코드를 처음 봤을 때 → 의심한 점 → 확인한 방법 → 결론" |

### pptx (PowerPoint)

| 원칙 | 적용 방법 |
|------|----------|
| slides 원칙 전체 | 위 slides 원칙을 동일하게 적용 |
| **양식 원칙** (Mayer) | 발표자 노트에 상세 설명 배치. 슬라이드 본문은 키워드/다이어그램 중심. 청중이 읽는 것과 발표자가 말하는 것을 분리 |

## 호기심 트리거 생성 규칙

**근거:** Loewenstein(1994) 정보 격차 이론 -- 사람은 "알고 있는 것"과 "알고 싶은 것" 사이의 격차를 인식할 때 호기심이 발생한다.

### 좋은 예

- "이 코드가 모든 테스트를 통과하는데, 어떤 조건에서 깨질까요?"
- "이 함수의 시간 복잡도가 O(n)으로 보이지만, 실제로는 O(n^2)인 이유가 뭘까요?"
- "이 싱글톤이 Thread-safe해 보이지만, 동시 접근에서 문제가 되는 이유는?"
- "이 컴포넌트가 deinit/dispose되지 않는데, 메모리 릭의 원인은 어디일까요?"

### 나쁜 예

- "왜 이럴까요?" -- 너무 막연하여 정보 격차를 생성하지 못함
- "이건 안 좋습니다" -- 질문이 아니라 판단문
- "어떻게 생각하세요?" -- 구체적 사실 기반이 아님
- "이 코드를 개선할 수 있을까요?" -- 격차가 아닌 당위를 제시

### 생성 규칙

1. **항상 구체적 사실 기반**: 코드의 특정 동작, 수치, 조건을 언급해야 한다.
2. **격차 크기 조절**: 너무 쉬운 질문(격차 없음)도, 너무 어려운 질문(격차 과대)도 피한다.
3. **답을 암시하되 완전히 드러내지 않는다**: "모든 테스트를 통과하는데"(정상 동작 암시) + "어떤 조건에서 깨질까요?"(미지 영역 지시).
4. **1개의 발견 사항에 1개의 트리거**: 여러 질문을 연쇄하지 않는다.
5. **Why 블록의 첫 문장으로 배치**: 트리거 직후에 답변이 오므로 인접 원칙을 유지한다.

## ZPD 보정 매트릭스

**근거:** Vygotsky(1978) 근접 발달 영역(Zone of Proximal Development) -- 학습자의 현재 수준과 도움을 받아 도달 가능한 수준 사이에서 교수가 가장 효과적이다.

| audience | 비계 수준 | Before/After/Why 조정 | 용어 |
|----------|----------|----------------------|------|
| **junior** | 높음 | Why를 상세히 설명. 기초 개념부터 단계적으로 풀어씀. After 코드에 주석을 풍부하게. "왜 이렇게 바꿨는지"를 한 줄씩 설명 | 용어 풀이 포함. 약어 사용 시 첫 등장에서 풀네임 병기 (예: "SRP(단일 책임 원칙)") |
| **mid** | 중간 | 표준 Before/After/Why 트리플 적용. 원칙 명명 + 핵심 영향 설명. After 코드에 핵심 변경만 주석 | 팀 공용 어휘 사용. 팀 내 합의된 용어는 설명 없이 사용 가능 |
| **senior** | 낮음 | Why를 **질문으로 전환** (답을 주지 않고 사고를 유도). Before/After만 제시하고 "이 변경의 트레이드오프는?"처럼 질문. 이미 알고 있을 기초 설명 완전 생략 | 축약/전문 용어 자유 사용. 학술 레퍼런스 직접 인용 가능 |

### 보정 규칙

- `audience` 파라미터가 지정되지 않으면 **mid**를 기본값으로 사용한다.
- 동일 출력 내에서 audience 수준을 혼합하지 않는다.
- junior에게 Praise(🟢)를 더 많이 배치하여 자신감을 강화한다 (최소 30%).
- senior에게 Insight(💡)를 더 많이 배치하여 깊은 토론을 유도한다 (최소 30%).

## 확장 원칙 (리서치 기반 강화)

아래 원칙은 2024-2026년 최신 연구에서 도출된 검증된 개선사항이다.

### 5. 인지 오류 유형 분류

> 각 발견 사항에 개발자가 **왜 그 실수를 했는지**를 인지심리학적으로 분류한다.

**근거:** Huang & Madeira(2024) HECR — 인지 오류 유형 분류를 적용한 코드 리뷰가 true positive를 ~400% 향상시킴.

| 태그 | 유형 | 의미 | 개선 방향 |
|------|------|------|----------|
| `[Slip]` | 실행 오류 | 올바른 지식이 있지만 실행에서 실수 (타이포, off-by-one) | 린팅, 체크리스트 |
| `[Rule]` | 규칙 오류 | 익숙한 패턴을 맞지 않는 상황에 적용 (API 계약 오해) | 전제 조건 학습 |
| `[Knowledge]` | 지식 부족 | 해당 개념을 접해본 적 없음 (race condition 모름) | 학습, 연습 |
| `[Lapse]` | 망각 | 알고 있지만 잊어버림 (에러 처리 누락) | 자동화, 리마인더 |

#### 적용 규칙

- 🔴 Must Fix, 🟡 Should Improve 발견에 오류 유형 태그를 부착한다.
- Why 블록에서 오류 유형에 따른 **맞춤 개선 방향**을 제시한다.
- "무엇이 틀렸는가"를 넘어 "왜 이런 실수가 발생했는가"를 다룬다.

---

### 6. 자기 설명 프롬프트 (Self-Explanation)

> Before 코드를 보여준 뒤, Why를 즉시 제공하지 않고 독자에게 먼저 생각할 기회를 준다.

**근거:** Chapagain & Rus(AAAI 2025) — 자기 설명이 코드 이해 깊이를 유의미하게 향상 (d=0.55). Utrecht University(2025) — 단계별 인출 연습이 1주 후 회상을 2.4배 향상.

#### 적용 규칙 (web/slides 포맷)

1. Before 코드를 ⚠️ 마커와 함께 제시한다.
2. **"이 코드의 문제가 무엇이고, 어떻게 고치시겠습니까?"** 질문을 배치한다.
3. 클릭/다음 슬라이드로 Why + After를 공개한다.

#### 적용 규칙 (md/wiki 포맷)

- 기존 Before → Why → After 순서를 유지하되, Why 블록 시작을 호기심 트리거 **질문**으로 하여 독자가 잠시 멈추고 생각하도록 유도한다.

---

### 7. 사양 근거 (Specification Grounding)

> 모든 발견 사항을 **구체적인 테스트, 계약, 사양**에 근거하여 기술한다.

**근거:** arXiv(2026) "The Specification as Quality Gate" — 사양 기반 리뷰가 개발자 수용률을 90.9% 향상. LLM 리뷰의 "에코 문제"(그럴듯하지만 틀린 발견)를 방지.

#### 적용 규칙

- 가능한 경우 위반된 테스트/계약/타입 시스템 제약을 명시한다.
  - 예: "이 코드는 `ThreadSafeQueue` 프로토콜의 thread-safety 계약을 위반합니다 (L42)" 
  - 나쁜 예: "이 코드는 thread-safe하지 않습니다"
- 테스트가 없는 경우, 어떤 테스트가 이 문제를 잡았을지 제안한다.
- 추측에 기반한 발견은 "확인 필요" 라벨을 붙인다.

### 8. 메타인지 비계 (Metacognitive Scaffolding)

> 리뷰를 읽는 개발자의 능동적 사고를 3단계로 유도한다.

**근거:** 다수의 2024-2025 연구 — AI 생성 콘텐츠는 "메타인지 게으름"을 유발할 위험. 계획-모니터링-평가 비계가 이를 방지.

| 단계 | 프롬프트 | 배치 위치 |
|------|---------|----------|
| **계획** | "이 코드를 읽기 전에, 어떤 유형의 문제를 예상하시나요?" | Layer 1 직후 (선택적) |
| **모니터링** | "이 설명이 예상과 일치하나요?" | 각 Why 블록 끝 (선택적) |
| **평가** | "이 수정이 작동하지 않는 경우는 없을까요?" | 각 After 블록 끝 (선택적) |

#### 적용 규칙

- 메타인지 프롬프트는 Layer 3 "생각해볼 점" 섹션의 Phase 6 질문에 **추가**된다.
- junior audience에서는 모니터링 프롬프트를 명시적으로 배치하여 자기 점검 습관을 형성한다.
- senior audience에서는 평가 프롬프트를 강화하여 비판적 사고를 유도한다.

### 9. 성장 마인드셋 프레이밍

> 모든 발견 사항을 실패가 아닌 **학습 기회**로 프레이밍한다.

**근거:** Dweck 마인드셋 이론 + Springer(2024) 코드 리뷰 불안 연구 — 인지행동 워크숍이 리뷰 불안을 유의미하게 감소시킴.

#### 적용 규칙

- 🔴 Must Fix에 **정상화 문구**를 포함한다: "이 패턴은 숙련된 개발자도 자주 빠지는 함정입니다"
- 결함 언어("실패", "잘못된") 대신 성장 언어("다음 단계", "마스터리 과정")를 사용한다.
- 반복 발견 시 퇴행이 아닌 **진전**을 강조한다: "이전에는 단순한 케이스에서 이 문제가 있었는데, 이번에는 더 복잡한 변형에서 발견되었습니다 — 단순 케이스는 이미 마스터하셨습니다"
