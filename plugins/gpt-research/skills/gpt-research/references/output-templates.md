# GPT-PRO Research Prompt 출력 템플릿

## 공통 헤더

모든 모드에서 프롬프트는 다음 헤더로 시작합니다:

```
# GPT-PRO Research Request
> Generated by Claude Code — gpt-research skill
> Mode: {module|arch|issue|custom}
> Date: {YYYY-MM-DD}
> Project: {프로젝트명}
```

---

## Role 섹션 — 모드별 템플릿

### module 모드

```
## Role
당신은 소프트웨어 아키텍트이자 코드 분석 전문가입니다.
아래 제공된 모듈의 소스 코드, 의존성, 인터페이스를 분석하고
요청된 리서치 질문에 답해주세요.
```

### arch 모드

```
## Role
당신은 소프트웨어 아키텍처 전문가입니다.
아래 제공된 프로젝트 구조, 의존성, 설정 파일을 기반으로
프로젝트의 아키텍처를 분석하고 요청된 리서치 질문에 답해주세요.
```

### issue 모드

```
## Role
당신은 디버깅 및 장애 분석 전문가입니다.
아래 제공된 에러 정보, 관련 소스 코드, git 히스토리를 분석하여
문제의 근본 원인을 파악하고 해결 방안을 제시해주세요.
```

### custom 모드

```
## Role
{사용자가 지정한 역할 또는 기본값: "당신은 소프트웨어 개발 전문가입니다."}
```

## Context 섹션 포맷

각 파일은 파일 경로 헤더와 코드 블록으로 구성합니다:

```
## Context

### 소스 파일

#### `src/auth/AuthManager.swift`
\```swift
// 파일 내용
\```

#### `src/auth/TokenStore.swift`
\```swift
// 파일 내용
\```

### 인터페이스/프로토콜

#### `src/protocols/Authenticatable.swift`
\```swift
// 파일 내용
\```

### 테스트

#### `tests/AuthManagerTests.swift`
\```swift
// 파일 내용
\```
```

**arch 모드의 Context에는 추가 하위 섹션:**

```
### 디렉토리 구조
\```
프로젝트/
├── src/
│   ├── auth/
│   ├── models/
│   └── ...
\```

### 프로젝트 문서
{CLAUDE.md, README.md 등의 내용}

### 의존성
{Package.swift, package.json 등의 내용}

### 빌드 시스템
{Makefile, Tuist 등의 내용}
```

**issue 모드의 Context에는 추가 하위 섹션:**

```
### 에러 정보
\```
{에러 메시지 또는 스택 트레이스}
\```

### Git 히스토리
\```
{관련 파일의 최근 커밋}
\```

### 환경 정보
- OS: {운영체제}
- Runtime: {런타임 버전}
- Dependencies: {관련 의존성 버전}
```

## Research Request 섹션

```
## Research Request

{구체적인 리서치 질문}

참고 사항:
- {추가 맥락이나 제약 조건}
- {특별히 주의할 점}
```

사용자가 명시적 질문을 제공하지 않은 경우, 모드별 기본 질문을 사용합니다:

| 모드 | 기본 리서치 질문 |
|------|-----------------|
| module | "이 모듈의 설계를 분석하고 개선점을 제안해주세요. 특히 책임 분리, 의존성 관리, 테스트 용이성 측면에서 검토해주세요." |
| arch | "이 프로젝트의 아키텍처를 분석하고 강점과 개선 기회를 식별해주세요. 확장성, 유지보수성, 관심사 분리 측면에서 평가해주세요." |
| issue | "제공된 에러 정보와 소스 코드를 분석하여 근본 원인을 파악하고, 구체적인 수정 방안을 코드 예시와 함께 제시해주세요." |
| custom | 사용자 직접 입력 (필수) |

## Expected Output 섹션

GPT-PRO의 응답은 Claude Code에 붙여넣어 바로 활용할 수 있도록 **공통 응답 envelope** + **모드별 본문**으로 구성합니다.

### 공통 응답 envelope

모든 모드에서 GPT-PRO에게 다음 응답 구조를 요청합니다:

```
## Expected Output

아래 형식을 **정확히** 따라 응답해주세요. 이 응답은 Claude Code에 다시 붙여넣어 후속 작업에 사용됩니다.

---

# GPT-PRO Research Response
> Mode: {module|arch|issue|custom}
> Confidence: {high|medium|low}

## Summary
{분석 결과를 3줄 이내로 요약. 핵심 결론을 먼저 쓰고, 근거를 이어서.}

## Findings

{모드별 분석 본문 — 아래 모드별 템플릿 참조}

## Code Suggestions

제안하는 코드 변경이 있다면, 각각 다음 형식으로 작성해주세요:

### CS-{번호}: {변경 제목}
- **File**: `{파일 경로}`
- **Lines**: {시작 라인}-{끝 라인} (알 수 있는 경우)
- **Type**: {add|modify|delete|refactor}

\```{language}
// 제안 코드
\```

> **이유**: {왜 이 변경이 필요한지 한 줄 설명}

## Action Items

후속 작업을 우선순위 순으로 정리해주세요:

- [ ] **[P0-Critical]** {설명} — `{관련 파일 경로}`
- [ ] **[P1-Important]** {설명} — `{관련 파일 경로}`
- [ ] **[P2-Suggestion]** {설명} — `{관련 파일 경로}`

## References

{참고한 문서, 라이브러리 문서, 관련 이슈 링크 등. 없으면 "N/A"}

---
```

### 필드 규칙

| 필드 | 필수 | 설명 |
|------|------|------|
| Summary | O | 3줄 이내 핵심 요약 |
| Findings | O | 모드별 분석 본문 (아래 참조) |
| Code Suggestions | △ | 코드 변경 제안이 있을 때만. 반드시 파일 경로 포함 |
| Action Items | O | 최소 1개. 우선순위 P0~P2로 분류 |
| References | △ | 외부 참고 자료가 있을 때만 |

### 우선순위 정의

| 레벨 | 의미 | 예시 |
|------|------|------|
| **P0-Critical** | 즉시 수정 필요 (크래시, 보안, 데이터 손실) | 메모리 릭, SQL 인젝션, 무한 루프 |
| **P1-Important** | 가까운 시일 내 수정 권장 (품질, 성능, 유지보수성) | 중복 코드, 느린 쿼리, 잘못된 추상화 |
| **P2-Suggestion** | 개선하면 좋은 사항 (스타일, 가독성, 미래 대비) | 네이밍, 문서화, 테스트 추가 |

## 청킹 시 헤더 템플릿

컨텍스트가 200K 문자를 초과하여 여러 프롬프트로 분할할 때:

### 첫 번째 청크

```
# GPT-PRO Research Request (Part 1/{총 파트 수})
> ⚠️ 이 리서치 요청은 {총 파트 수}개로 분할되어 있습니다.
> 모든 파트를 읽은 후 마지막 파트의 Research Request에 답해주세요.

## Role
{역할}

## Context (Part 1)
{첫 번째 청크의 컨텍스트}

---
> ▶ 다음 파트를 붙여넣어 주세요.
```

### 중간 청크

```
# GPT-PRO Research Request (Part {N}/{총 파트 수})
> 이전 파트의 맥락을 유지하면서 읽어주세요.

## Context (Part {N}, continued)
{N번째 청크의 컨텍스트}

---
> ▶ 다음 파트를 붙여넣어 주세요.
```

### 마지막 청크

```
# GPT-PRO Research Request (Part {N}/{총 파트 수} — Final)
> 이전 파트의 모든 맥락을 종합하여 아래 질문에 답해주세요.

## Context (Part {N}, final)
{마지막 청크의 컨텍스트}

## Research Request
{리서치 질문}

## Expected Output
{기대 출력 형식}
```
