# Report Template — Top-3 매칭 리포트 구조

Phase 4에서 생성하는 리포트의 정확한 섹션 구조. 누락 없이 모든 섹션을 포함하라. **섹션 3의 토큰 발췌는 `references/designs/{slug}.md` 본문에서 직접 인용**하며, 의역·요약하지 말라.

## 출력 원칙

- **마크다운 텍스트**로 출력하고, 사용자가 파일 저장을 원하면 경로를 묻는다.
- 리포트 총 분량은 600~1,200단어 범위. 장황함은 정보 가치를 해친다.
- 데이터가 없는 섹션은 `(데이터 없음)`으로 명시. 비워두지 말라.

## 3. Top-3 디자인 토큰 발췌 (본문 직접 인용)

각 브랜드의 `references/designs/{slug}.md`에서 Section 2(Color Palette) / Section 3(Typography) / 핵심 수치(shadow·radius·spacing)를 **원문 그대로** 발췌. 의역·요약 금지.

### 🥇 {Brand-1} 핵심 토큰

```

{예: "Deep Navy (#061b31): --hds-color-heading-solid. Primary heading color"}
...
```

**Typography** (본문 Section 3 직접 인용):
```
{예: "sohne-var with OpenType ss01 stylistic set enabled"}

{예: "Negative letter-spacing: -1.4px at 56px"}
...
```

```
{예: "Shadow: rgba(50,50,93,0.25) multi-layer"}
{예: "Border-radius: 4px-8px (conservative)"}

## 6. 주의사항

- 본 리포트는 awesome-design-md DESIGN.md 본문(getdesign@0.6.8 스냅샷)을 직접 인용했다. 상류가 갱신되면 `scripts/fetch-catalog-diff.sh`로 변경을 감지할 것.
- 섹션 3의 토큰은 본문에서 원문 그대로 발췌되었으며 해당 브랜드의 공개 CSS 값에 근거한다. 각 브랜드 상표·로고는 해당 회사 소유이며 상업 사용 전 법적 검토 필요.
- trait 벡터의 신뢰도가 `low`인 필드는 추론 포함이 많다. 실제 적용 전 사용자 재검토 필요.

## 작성 시 체크리스트

리포트를 출력하기 전 아래를 자체 검증하라:

- [ ] 섹션 1~7 모두 존재 (이전 6섹션 → 본문 발췌 섹션 추가)
- [ ] Top-3 각 항목에 Match/Gain/Loss 3문장 포함
- [ ] **섹션 3 토큰 발췌를 `references/designs/{slug}.md`에서 직접 인용** (의역·요약 없이)
- [ ] 발췌한 hex·폰트·수치가 본문과 글자 수준으로 일치
- [ ] Contrast Reference 포함 (Top-3와 방향 반대)
- [ ] URL 슬러그에 점 포함된 경우 정확히 표기 (`linear.app`, `x.ai`)
- [ ] 본문 로드 경로(`Read ...`)와 설치 커맨드 모두 표기
- [ ] 주의사항에 스냅샷 시점·상표 고지 명시

체크리스트 하나라도 빠지면 리포트를 재작성한다.

## 카테고리·무드 불일치가 심할 때의 포맷

전체 후보 최고 점수가 `< 0.4`이면 템플릿 2번 섹션을 다음으로 대체:

```markdown
## 2. 카탈로그 매칭 실패

현재 trait 벡터에 대해 카탈로그 내 강한 매칭(>= 0.4)이 없습니다.

**원인 추정**:
- {추정 1}
- {추정 2}

**대안**:
1. **범위 확장**: `--platform any`로 재실행, 또는 category 제약 해제
2. **디자이너 기반 접근**: `design-research` 스킬로 Dieter Rams / Jony Ive / Mondrian 등 디자이너 토큰 생성
3. **직접 정의**: 프로젝트 루트에 자체 DESIGN.md 작성
```

이 경우 섹션 3~6은 "대안 경로 안내"로 축소한다.
